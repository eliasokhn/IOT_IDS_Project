SECTION 1 — WHAT THIS PROJECT IS
================================================================================

We are building an Intrusion Detection System (IDS) for IoT network traffic.
The dataset is CICIoT2023 from the University of New Brunswick — 45 million
rows of network flow records, each labeled as either normal (BENIGN) or one
of 33 different attack types.

A "network flow" is a summary of a conversation between two devices on a
network. Instead of capturing every individual packet, a flow captures
statistics like: how many bytes were sent, how fast, how long the connection
lasted, etc. We have 39 such numeric features per flow.

We train two types of models:
- Logistic Regression (LR): a simple, linear model. Think of it as drawing
  straight lines to separate classes. Fast to train, but limited in what it
  can learn.
- Gradient Boosting (GB): a powerful ensemble model. It builds hundreds of
  small decision trees one after another, each one correcting the errors of
  the previous one. Much more powerful than LR for complex problems.

We train each model on three tasks simultaneously:
- Binary: just BENIGN vs MALICIOUS (2 classes)
- 8-class: BENIGN + 7 attack families (DDoS, DoS, Mirai, etc.)
- 34-class: BENIGN + all 33 individual attack types


KEY TERMS YOU NEED TO KNOW:

Macro F1 Score — our PRIMARY metric.
  F1 is a balance between precision (when we say "attack", are we right?) and
  recall (of all real attacks, how many did we catch?). "Macro" means we
  compute F1 separately for each class and then average them, treating all
  classes equally regardless of how many samples they have. This is important
  because we have tiny classes like XSS (1111 samples) and huge ones like
  DDoS-ICMP_FLOOD (2 million samples). Without macro averaging, the model
  could ignore XSS entirely and still look good on paper.

FPR Benign (False Positive Rate on Benign traffic) — the % of normal
  (BENIGN) network flows that the model incorrectly flags as attacks.
  This is critical for a real IDS: if you flag legitimate traffic as an
  attack, you block normal users. Lower is better. 0% = perfect, 100% = the
  model calls every benign packet an attack.

Class Imbalance — our dataset has 5776x more data for the most common attack
  type (DDoS-ICMP_FLOOD) than for the rarest attack type (UPLOADING_ATTACK).
  BENIGN itself is only 2.34% of the data. If you train without correcting for
  this, the model just learns to always predict "DDoS-ICMP_FLOOD" and ignores
  rare classes entirely.

class_weight='balanced' — a technique to fix class imbalance. Instead of
  treating all training samples equally, we give higher weight (importance)
  to samples from rare classes. A class with 358 samples gets ~37x more weight
  than a class with 2 million samples during training.

Sample Fraction (sample_frac) — because 45 million rows is too large to
  process on a laptop, we sample a percentage of each class. At 25%, we get
  ~11 million rows. At 30%, we get ~13.5 million rows.

Deduplication — about 47% of rows in this dataset are exact duplicates or
  near-duplicates (same features, different label). We remove them BEFORE
  splitting into train/test sets to avoid data leakage (the same row appearing
  in both training and testing, which would make results artificially good).

RobustScaler — a method to normalize feature values so they are all on a
  similar scale. It subtracts the median and divides by the IQR (interquartile
  range = the range covering the middle 50% of values). It is more resistant
  to extreme outliers than the standard approach of using mean and std deviation.

GPU vs CPU — training on a GPU (graphics card) uses thousands of parallel
  cores to speed up computation. For Gradient Boosting on 4.8M training
  samples, GPU is 5-10x faster than CPU. Our GPU is NVIDIA RTX 4060 Laptop
  with 8GB VRAM.

SECTION 2 — CHANGE 1: SAMPLE FRACTION 25% → 30%
================================================================================

WHAT WE CHANGED:
  configs/data_config.yaml: sample_frac 0.25 → 0.30
  run_pipeline.py: default --sample-frac 0.25 → 0.30

WHY:
  More training data generally improves model generalization, especially for
  rare classes. At 25%, rare classes like XSS had only ~925 training samples.
  At 30%, they get ~1111. This helps the model learn the decision boundary for
  those rare classes better.

  The trade-off is longer training time and more memory usage.

RESULT:
  Sampled total went from ~11M to ~13.5M rows. After deduplication, training
  set grew from ~4.1M to ~4.8M samples.

SECTION 3 — PROBLEM 1: LIGHTGBM GPU CRASHES
================================================================================

WHAT HAPPENED:
  Every time GB tried to train on GPU using LightGBM, it crashed with:
  "[LightGBM] [Fatal] Check failed: (best_split_info.left_count) > (0)"

WHY IT CRASHES:
  LightGBM on GPU uses a "histogram" algorithm to find the best split point
  for each decision tree node. On CPU it uses 255 bins (buckets) to
  approximate the data distribution. On GPU it uses only 63 bins because the
  GPU memory architecture works better with smaller histograms.

  With 63 bins instead of 255, the approximation is coarser. Combined with
  our extreme class imbalance (5776x ratio) and the fact that we use sample
  weights to compensate for that imbalance, the GPU sometimes finds a "split"
  where all data points fall on one side — leaving the other child node with
  zero samples. LightGBM has an internal safety check that says "a split must
  put at least one sample on each side", and when that check fails, it crashes.

  This is a known bug in LightGBM's GPU implementation. It does not happen
  on CPU because the 255-bin approximation is precise enough to avoid these
  degenerate splits.

THE FIX:
  File: src/models/train_gb.py

  Before our fix, the fallback chain was:
    LightGBM GPU → fails → CPU sklearn HistGBM

  After our fix:
    LightGBM GPU → fails → XGBoost GPU → (if that also fails) → CPU sklearn

  XGBoost is a different Gradient Boosting library with a more robust GPU
  histogram implementation that handles imbalanced weighted data correctly.
  We also updated the log message from the confusing "Falling back to CPU"
  to the accurate "LightGBM GPU failed — trying XGBoost GPU next."

RESULT:
  All three GB tasks (binary, 8class, 34class) now train on XGBoost GPU.
  Speed improvement vs CPU sklearn:
    Binary: 66 seconds → 8 seconds (8x faster)
    8-class: 296 seconds → 64 seconds (4.6x faster)
    34-class: 590 seconds → 283 seconds (2x faster)

SECTION 4 — PROBLEM 2: LOGISTIC REGRESSION COMPLETELY BROKEN
================================================================================

WHAT HAPPENED:
  LR results were catastrophically bad:
    Binary:   Macro F1 = 0.39, FPR Benign = 5.45%, Accuracy = 48.96%
    8-class:  Macro F1 = 0.19, FPR Benign = 89.52%
    34-class: Macro F1 = 0.12, FPR Benign = 100.00%

  FPR Benign = 100% means the model classified EVERY SINGLE benign flow as
  an attack. The model was completely useless.

WHY IT HAPPENED — A CHAIN OF THREE PROBLEMS:

  Step 1: Infinity values in the data.
    The "Rate" feature (packets per second) had 322 rows with infinite values.
    This happens when the time duration of a flow is exactly zero — division
    by zero gives infinity. Our pipeline replaces infinity with the column's
    maximum finite value (inf_fill_strategy: "col_max"). For Rate, this maximum
    was approximately 100,000,000 (1e8).

  Step 2: RobustScaler did not fix these extreme values.
    RobustScaler works by: (value - median) / IQR.
    For the Rate feature, most flows have a rate near zero, so the IQR
    (the range covering the middle 50% of values) is also very small — maybe
    1 or 2. When you divide 1e8 (the replaced infinity value) by an IQR of 1,
    you still get 1e8. The scaler did nothing to the outlier.
    This was confirmed by the runtime warning: "max abs = 4.76e+08 after scaling".

  Step 3: The optimizer (SAGA) diverged because of this.
    Logistic Regression trains by gradient descent — it iteratively adjusts
    its internal weights to reduce classification errors. The gradient (the
    adjustment signal) for each training sample is proportional to that
    sample's feature values. A sample with Rate = 4.76e+08 produces a gradient
    update that is 476 million times larger than a normal sample. The SAGA
    optimizer cannot recover from this — the weights oscillate wildly and
    never converge to a good solution. The model ends up in a degenerate state
    where it predicts everything as malicious.

THE FIX:
  File: src/features/preprocessing.py

  We added a post-scaling clipping step. After RobustScaler transforms the
  data, we clip every feature value to the range [-20, +20].

  In RobustScaler's output units, ±20 means "20 IQR ranges away from the
  median". In any real network traffic distribution, a legitimate feature
  value will never be 20 IQRs from the median. The only values that exceed
  this are the pathological infinity→column-max replacements. Clipping them
  to 20 eliminates the problem while preserving all real signal.

  This fix does NOT affect Gradient Boosting. Decision trees split data based
  on thresholds (e.g., "Rate > 5?") and are completely immune to the scale of
  feature values — whether Rate is 20 or 4.76e+08 doesn't change which branch
  a sample goes to, as long as the relative ordering is preserved.

RESULT AFTER FIX:
    Binary:   Macro F1 = 0.39 → 0.83, FPR = 5.45% → 0.16%
    8-class:  Macro F1 = 0.19 → 0.53, FPR = 89.52% → 39.53%
    34-class: Macro F1 = 0.12 → 0.50, FPR = 100% → 97.10%

  Binary went from broken to actually useful. 8-class and 34-class improved
  significantly but still have high FPR — explained in Section 6.

SECTION 5 — PROBLEM 3: LOGISTIC REGRESSION STUCK / TOO SLOW
================================================================================

WHAT HAPPENED:
  After the clipping fix, LR binary ran for 15-20 minutes without finishing,
  then got stuck again. Previously it finished in 34 seconds.

WHY:
  The 34-second run before was deceptively fast — the model was NOT actually
  learning. The extreme gradient from the 4.76e+08 outlier was pushing the
  weights to a fixed degenerate point in just 1-2 iterations. It looked fast
  because it converged instantly to the wrong answer.

  After fixing the clipping, the model now has to do real gradient descent with
  proper gradients. The solver we were using (SAGA) processes training samples
  one at a time in a stochastic manner. With 4.8 million training samples and
  max_iter=1000 (meaning up to 1000 full passes over the data), this is an
  enormous amount of computation on a single CPU core.

  Additionally, since sklearn version 1.8, the n_jobs=-1 parameter (which was
  supposed to use all CPU cores) has NO effect on the SAGA solver. So we were
  running single-threaded on 4.8M samples × 1000 iterations.

THE FIX:
  Two changes to configs/model_config.yaml:

  1. Changed solver from "saga" to "lbfgs".
     L-BFGS is a quasi-Newton optimization method that uses information about
     the curvature of the loss function (second-order information) to take
     smarter steps. SAGA is first-order (it only uses gradient direction).
     L-BFGS converges in far fewer iterations on well-scaled data.
     Crucially, lbfgs IS multi-threaded through numpy/BLAS linear algebra
     libraries — it uses all CPU cores without needing n_jobs.

  2. Reduced max_iter from 1000 to 300.
     With properly scaled data and lbfgs, 300 iterations is sufficient for
     most tasks. The model still hit max_iter on all 3 tasks (meaning it
     could improve more with more iterations), but the results were already
     dramatically better than before, and each task now finishes in under
     11 minutes total instead of hanging forever.

RESULT:
  LR binary finished in 66 seconds instead of hanging.
  LR 8-class finished in 290 seconds.
  LR 34-class finished in 652 seconds.

SECTION 6 — GB HYPERPARAMETER TUNING
================================================================================

After getting XGBoost GPU working, we tried to improve GB results by tuning
its configuration parameters.

CHANGES TO configs/model_config.yaml:

  max_iter: 300 → 1000
    The maximum number of trees to build. XGBoost was still improving at
    iteration 299 in the previous run — we gave it room to run longer.
    Early stopping would prevent it from actually building 1000 trees if
    it stopped improving.

  learning_rate: 0.1 → 0.05
    How much each new tree corrects the errors of the previous trees.
    A smaller learning rate means more conservative corrections — the model
    learns more carefully and generalizes better, but needs more trees.
    Lower learning rate + more trees is a well-known recipe for better
    Gradient Boosting performance.

  max_depth: 6 → 8
    How deep each decision tree can grow. Deeper trees can capture more
    complex relationships between features. From 6 levels to 8 levels means
    each tree can make up to 256 decisions instead of 64.

  min_samples_leaf: 20 → 10
    The minimum number of training samples required in a leaf node (terminal
    node of a tree). Lower values allow the tree to make finer distinctions,
    which is important for rare classes with only a few hundred samples.

  subsample: 0.8 (new)
    Each tree is trained on a random 80% of the training data instead of
    all of it. This introduces randomness that helps prevent overfitting
    (memorizing training data instead of learning general patterns).

  colsample_bytree: 0.8 (new)
    Each tree uses a random 80% of the features. Same idea as subsample —
    the randomness prevents any single feature from dominating all trees.

  n_iter_no_change: 20 → 30
    Early stopping patience. Stop training if validation loss does not
    improve for 30 consecutive trees. We increased this so the model has
    more patience before giving up.

BUG FIXED IN XGBoost EARLY STOPPING:
  File: src/models/train_gb.py

  Before our fix, XGBoost was evaluating early stopping by measuring loss on
  the TRAINING data itself. Training loss always decreases as you add more
  trees (the model memorizes training data better and better). So early
  stopping never triggered — the model always ran all max_iter trees.

  After our fix, XGBoost holds out 10% of training data as an internal
  validation set. It measures loss on THIS held-out data (which the model
  has never seen during tree building). When the validation loss stops
  improving, training stops. This is proper early stopping.

RESULT:
  The tuning barely improved results:
    Binary:   0.8832 → 0.8879 F1  (marginal gain)
    8-class:  0.6393 → 0.6448 F1  (marginal gain)
    34-class: 0.5855 → 0.5846 F1  (actually slightly worse!)

  The early stopping triggered early: binary at 538 trees, 8-class at 295,
  34-class at only 213. This means the model was not benefiting from more
  trees — it hit its generalization ceiling quickly.

  WHY TUNING DIDN'T HELP — ROOT CAUSE:
    The bottleneck is the DATA, not the model configuration. Specifically:
    1. After deduplication, the test set has only ~138 BENIGN samples because
       BENIGN is just 2.34% of the dataset.
    2. BENIGN network traffic shares many feature characteristics with some
       attack types. The 43 features we have are not always enough to perfectly
       separate benign from every one of 33 attack types.
    3. For the 34-class task, the model must distinguish between 34 very
       similar classes using features designed to catch attacks, not designed
       to preserve benign traffic uniqueness.
    No hyperparameter tuning can fix a problem that lives in the data itself.

SECTION 8 — THE STREAMING SIMULATION (WHY 92% ALERT RATE IS NOT A BUG)
================================================================================

After training, the pipeline runs a "streaming simulation" — it feeds batches
of 500 records through the binary model and monitors the alert rate (% predicted
as attack) and feature drift (unusual feature value changes).

The simulation showed 90-97% alert rate even in the "normal" scenario.
This looks alarming but it is NOT a bug. Here is why:

The simulation samples from the TEST SET, which has the same class distribution
as the full dataset: ~97.66% of records are attacks and only ~2.34% are benign.
So feeding test set records to the model should indeed produce ~92% alert rate —
that is the correct answer given the data.

The scenario names ("normal", "attack_surge", "gradual_drift") refer to
different sampling strategies and feature perturbations applied to the test
data, not to "this is a normal day with low attack traffic."

The "FEATURE DRIFT" warning for Rate = inf in batch 16 is also expected — it
appears in every run at the same batch because the random sampling always hits
the same row with an infinity Rate value.

SECTION 9 — WHAT CAN BE DONE TO IMPROVE
================================================================================

1. INCREASE SAMPLE FRACTION TO 50%
   Currently at 30% (~13.5M rows → ~4.8M train after dedup). At 50%, you get
   ~22.5M rows → ~8M train samples. More data especially helps the rare classes
   (XSS, BruteForce, etc.) and would likely push Macro F1 up by 3-5%.
   Trade-off: deduplication step (currently 10 minutes) would take 15-20
   minutes, and training would take longer.

2. FEATURE ENGINEERING
   The current 43 features are raw network flow statistics. Creating new
   features from combinations of existing ones (e.g., bytes_per_packet =
   total_bytes / packet_count, or ratio of forward/backward traffic) could
   make class boundaries cleaner and help especially for 34-class separation.

3. FIX LOGISTIC REGRESSION CONVERGENCE
   LR still hits max_iter=300 on all tasks (it shows a ConvergenceWarning).
   Increasing max_iter to 500-1000 would let it fully converge. The trade-off
   is training time — 8-class LR already takes 5 minutes at 300 iterations.
   Since LR is just a baseline and GB is the real model, this is low priority.

4. INSTALL cuML (RAPIDS) FOR GPU LOGISTIC REGRESSION
   cuML is NVIDIA's GPU-accelerated machine learning library. It would allow
   LR to train on GPU, reducing LR training time from minutes to seconds.
   Installation is complex (requires specific CUDA + Python version combination)
   and gives diminishing returns since LR is a baseline.

5. ADDRESS THE 34-CLASS FPR PROBLEM
   The 81% FPR on benign for 34-class is a data distribution problem. Options:
   a) SMOTE (Synthetic Minority Over-sampling Technique): artificially generate
      more synthetic BENIGN samples to give the model more to learn from.
   b) Use a two-stage classifier: first predict binary (benign vs attack), then
      for samples predicted as attack, run the 34-class model. This way the
      34-class model never needs to predict "benign" — the binary model handles
      that with 0.43% FPR.
   c) Accept it: on a real network, 34-class classification is used primarily
      to categorize attacks that were already flagged by the binary model, not
      to distinguish attacks from benign traffic.

6. REMOVE LIGHTGBM FROM THE GPU PATH
   LightGBM GPU crashes every single run (the empty-node split bug). The code
   still tries LightGBM first every time, wasting ~5-10 seconds per task before
   failing over to XGBoost. Removing LightGBM from the GPU detection logic
   would save time and eliminate the confusing error messages.
